# v14.10 Node FED_SWARM_SCHEDULE Ready-DAG Parity

状态：complete

协议标签：`v14.10-protocol`

## What changed

Node now accepts `FED_SWARM_SCHEDULE` and matches the existing Go `scheduleSwarmSteps` contract. Before executing a step, the gateway validates the full signed Swarm graph, topologically orders dependency-ready steps, and uses original input order as the deterministic tie rule.

Execution remains serial. Every scheduled step reuses the existing signed task verification, dependency artifact and upstream receipt binding, micro-contract, one-step migration, conflict resolution, plan digest propagation, signed close, and audit paths.

## Close evidence

A successful scheduled Swarm signs exactly:

```json
{"scheduler":{"mode":"ready-dag","step_order":["summary","followup"]}}
```

`step_order` records the actual serial execution order. Node `verifySwarmClose` accepts this optional backward-compatible evidence only when `mode` is `ready-dag` and every `step_order` entry matches the corresponding ordered `step_receipts` entry, with no duplicates or missing steps. Normal `FED_SWARM_OPEN` remains input-order and does not add `scheduler`.

## Fail-closed graph validation

The scheduler rejects missing dependencies, duplicate step IDs, self-dependencies, and a cycle or otherwise unresolvable graph before any `FED_RECEIPT` or `FED_SWARM_CLOSE` is emitted. This preflight prevents a malformed graph from partially executing.

## Honest boundary

This slice adds deterministic ready-DAG ordering only. It adds no parallel execution, no resource scheduling, no economic ranking, no LLM scheduling, and no new trust inputs. It does not decompose intent, select workers through a new ranking system, or weaken existing Swarm verification.
