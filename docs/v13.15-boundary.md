# v13.15 Boundary: Node Receipt Checkpoint Verification

状态：complete

目标：Close the local verifier gap where Node `FED_RECEIPT` verification accepted receipt-carried checkpoint evidence without checking the checkpoint ref list, parent chain, task binding, or worker signature.

## What Landed

- `verifyFederatedReceipt` now verifies optional receipt checkpoint evidence after the receipt signature and artifact manifest checks.
- `checkpoint_refs` must be a list of non-empty strings, and `checkpoints` must be a list of objects when either field is present.
- The checkpoint list count must match `checkpoint_refs`, each checkpoint must bind the same `task_id`, each `checkpoint_id` must match the corresponding ref, and the checkpoint parent chain starts at `receipt.resumed_from` when present or `null` for a fresh task.
- Each checkpoint must carry a valid worker `checkpoint_signature` over the checkpoint body without `checkpoint_signature`.
- `asp-verify.mjs fed-receipt <frame.json> <trusted-zones.json> [task.json]` inherits the same fail-closed checkpoint evidence verification.
- `test-vectors.test.mjs` covers accepted signed checkpoint evidence, ref mismatch rejection, checkpoint signature mismatch rejection, and CLI acceptance/rejection.

## Non-Goals

- No model KV/cache restore.
- No new checkpoint storage API.
- No external task store lookup.
- No scheduler, upper-layer demo/master-agent orchestration, or A2A/ARD compatibility.
- No hardware remote attestation or container namespace sandbox claim.

## Why This Boundary

Ultimate sections 5-6 require multi-signal routing and Swarm lifecycle decisions to stay accountable across state sync, failure migration, output verification, receipt closure, and reputation updates. Go already carried signed checkpoint evidence and resume parent links, but the Node verifier did not verify receipt-carried checkpoint evidence. v13.15 closes that local proof gap without adding new orchestration or external infrastructure.

The boundary stays deliberately narrow: it verifies evidence already present on a signed receipt. It does not claim real model-state restoration, distributed resume, or cross-node checkpoint availability.

## Test Contract

The tests must fail if:

- Node accepts a `FED_RECEIPT` whose `checkpoint_refs` do not match the signed `checkpoints` list.
- Node accepts a checkpoint whose body was changed after `checkpoint_signature` was created.
- `asp-verify.mjs fed-receipt` accepts the same malformed checkpoint evidence.
- Public docs describe this as model KV restore, scheduler orchestration, A2A/ARD compatibility, or external checkpoint storage instead of local receipt checkpoint evidence verification.
