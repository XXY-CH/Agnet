## What changed

The Darwin Apple `container` adapter now creates one agent-owned private `0700` workspace directory per run, stages caller input below `workspace/input`, pre-creates `workspace/result`, and mounts that directory read-write exactly once at `/work`.

It no longer uses Apple-side copy operations, a temporary `/work` filesystem, or separate input/result mounts. The container remains constrained with a read-only root filesystem, UID/GID `65532:65532`, `--network none`, `--no-dns`, `--cap-drop ALL`, requested CPU/memory limits, and `nofile=64:64` plus `nproc=64:64` ulimits.

Before launch the adapter pins workspace device, inode, owner UID, and mode. After execution it reopens and revalidates the same directory, permits only the staged `input` tree and `result`, rejects symlinks, hardlinks, devices, unexpected paths, escaping paths, and aggregate/result size overflow, then opens `result` without following links and reads it bounded.

## Honest boundary

This is a real Apple Container private-workspace mount proof, not full Docker isolation parity. It does not claim an Apple tmpfs quota, filesystem-size quota equivalent to Docker tmpfs, or PID parity (including PID namespace parity) beyond the applied `nproc` ulimit. It does not claim host-kernel equivalence, hardware attestation, or general-purpose sandbox completeness.
