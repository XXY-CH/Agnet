# v13.7 Boundary: Signed Sandbox Attestation Verifier

状态：complete

目标：把 v13.6 local sandbox proof verifier 推进到独立 signed attestation evidence verifier, without claiming hardware attestation or real container isolation.

## What Landed

- `asp-verify.mjs sandbox-attestation <frame.json> <trusted-zones.json> <attestation.json> <trusted-attestors.json>`.
- It verifies the trusted `FED_RECEIPT` and embedded `local.sandbox.v1` proof first.
- It verifies `asp-sandbox-attestation/v1` signed evidence from a descriptor with `sandbox.attest`.
- It requires the attestor to be pinned in a caller-supplied trusted attestor file.
- It binds attestation evidence to:
  - receipt digest
  - task id
  - sandbox digest
  - sandbox claim
  - policy digest
  - sandbox class
  - runtime identity
  - observed timestamp
- It rejects stale, future-dated, mismatched, unsigned, untrusted-attestor, and missing-capability evidence.
- Verifier output includes `hardware_attestation: false`.

## Non-Goals

- This is not hardware remote attestation, not container namespace execution, and not a TEE quote.
- It does not make `container-namespace` supported.
- It does not run tools inside a remote or stronger runtime.
- It does not trust attestation evidence without an explicit trusted attestor file.
- It does not close the hosted external-host reachability gate.

## Why This Boundary

Ultimate needs remote-verifiable execution claims. v13.6 made local sandbox proof class verifier-owned; v13.7 adds the next proof boundary: an independent signed attestor can make an attestation statement, and the verifier can check that statement is fresh, trusted, and bound to the exact receipt and sandbox evidence.

That still stops short of hardware attestation. A future slice needs real runtime evidence, such as a VM/container/TEE quote or comparable external runtime proof.

## Test Contract

The tests must fail if:

- `sandbox-attestation` accepts attestation evidence without first verifying the receipt and sandbox proof.
- Attestation evidence is not bound to the same receipt digest.
- Attestation evidence is not bound to the same sandbox digest.
- The attestor is not trusted or lacks `sandbox.attest`.
- The evidence is stale or future-dated.
- The docs describe this as hardware attestation.
