# v14.8 Boundary — Swarm Conflict Resolution

状态：complete
协议标签：`v14.8-protocol`

## What changed

v14.8 adds deterministic Swarm conflict resolution for the local-first proof surface. When two or more Swarm steps write the same artifact ref and the resulting artifact manifests carry different SHA-256 digests, Node and Go gateways create signed `conflict_resolutions` evidence and include it in `FED_SWARM_CLOSE.close`.

Each conflict resolution entry binds:

- `swarm_id`
- `artifact_ref`
- `candidate_step_ids`
- `chosen_step_id`
- `chosen_worker`
- `reason`
- `resolution_digest`
- Zone `signature`

The rule is intentionally boring and deterministic: choose the candidate from the worker with higher agent_score reputation; if scores tie, choose alias ascending and record `reason: "alias_tiebreak"`.

## Verifier boundary

Node `verifySwarmClose` treats `close.conflict_resolutions` as optional and backward-compatible. When present, it verifies the signed entry against the close Zone key, recomputes `resolution_digest` over the canonical entry body without digest/signature fields, checks candidate steps are distinct and present in close step receipts, checks the chosen step is one of the candidates, and checks `chosen_worker` matches the chosen step receipt worker descriptor.

## Non-goals

- No voting/quorum.
- No automatic merge of conflicting content.
- No payment/settlement.
- Still local-first: no global arbitration service, remote reputation oracle, production overlay, or public marketplace.

## Evidence

- Node gateway conflict test covers higher agent_score reputation selection and tampered signature rejection.
- Go gateway conflict test covers Go-produced `conflict_resolutions` accepted by Node `verifySwarmClose`.
- Test vector coverage rejects a conflict resolution whose candidate step is absent from the close step receipts.
