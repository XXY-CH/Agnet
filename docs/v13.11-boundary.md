# v13.11 Boundary: Audit-Backed Receipt-Count Reputation

状态：complete

目标：Replace semantic discovery reputation's hardcoded completed-receipt demo value with counts read from the persisted audit log in both Node and Go.

## What Landed

- Node federation gateway reputation now counts completed `fed_receipt` audit records for each worker AID from the persisted audit log before ranking query results.
- Go FED_QUERY reputation now counts completed `go_fed_receipt` audit records for each worker AID from `f.Audit.Path` before ranking query results.
- Missing, empty, unreadable, or partially malformed audit logs fail safe to zero counted receipts instead of inventing reputation.
- Semantic discovery still returns inspectable `discovery_evidence.reputation.completed_receipts` and `ranking.reasons`, but receipt counts come from the persisted audit log.
- `test/federation-gateway.test.mjs`, `test/go-fed-discovery.test.mjs`, and `test/docs-contract.test.mjs` cover the audit-backed receipt-count boundary.

## Non-Goals

- No cross-session ML.
- No global graph.
- No third-party reputation oracle.
- No global reputation coin.
- No vector database.
- No public marketplace.
- No scheduler integration.

## Why This Boundary

v13.4 and v13.10 made semantic discovery evidence-first, but the receipt-count reputation input was still a fixed demo value. That made the ranking surface inspectable but not evidence-backed.

v13.11 closes that gap by reading completed receipt records from the same persisted hash-chained audit log used for proof and receipt verification. Reputation remains a narrow local discovery input: receipt counts come from the persisted audit log, not a hardcoded demo value, not cross-session ML, not a global reputation oracle.

## Test Contract

The tests must fail if:

- Node or Go query results use a fixed completed receipt count instead of audit records.
- Completed receipt records for a different worker increase the queried worker's reputation count.
- Missing or malformed audit log lines crash query ranking instead of failing safe.
- Public docs describe the receipt count as temporary in-memory state, a hardcoded demo value, cross-session ML, or a global reputation oracle.
